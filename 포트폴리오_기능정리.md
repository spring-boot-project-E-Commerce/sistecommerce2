# 쇼핑몰 팀 프로젝트 — 내 담당 파트 정리

> Spring Boot 3.x · Java 21 · Oracle DB · JPA/Hibernate · Thymeleaf

---

## 1. 장바구니 (Cart 패키지)

패키지를 신규로 분리해서 엔티티부터 API까지 전체 구현.

**엔티티 설계**

- `Cart`: member(FK), options(FK), quantity, createdDate(`LocalDate`) — DDL의 `DATE` 타입에 맞춰 `LocalDateTime`이 아닌 `LocalDate` 사용
- `CartLog`: 장바구니 변동 이력 기록, actionDate는 `LocalDateTime` (`TIMESTAMP` 타입)
- `@ManyToOne(fetch = LAZY)` + `@JoinColumn`으로 연관관계 설정

**서비스 설계**

- `addCart()`: `@Transactional` 필수 — 없으면 `save()`가 커밋되지 않음
- `getReferenceById()` 활용: member, options 조회 시 SELECT 없이 프록시 참조만 사용해 FK 세팅
- `list()`: `findByMember_Seq()` (JPA 메서드 네이밍 — 연관 엔티티 필드명 기준) → `CartDto`로 변환 반환
- `CartDto`에 `getItemTotal()` 계산 메서드 포함 (`(price + additionalPrice) * quantity`)

**컨트롤러 분리**

- `CartViewController` (`@Controller`): 장바구니 목록 페이지 렌더링
- `CartApiController` (`@RestController`): 담기/삭제/수량변경 REST API
- `@AuthenticationPrincipal CustomUserDetails`로 memberSeq 직접 주입 — 클라이언트에서 memberSeq를 받지 않음 (보안)

**프론트 연동**

- 상품 상세 페이지(`product/detail.html`)에서 옵션 선택 후 Fetch API로 `POST /api/cart` 비동기 호출
- `id="optionsSeq"`, `id="quantity"` ID 통일 — HTML 필드명을 Java/DB 기준에 맞춤

## 2. 마이페이지 — 배송지 관리

**백엔드 구조**

서비스는 `member` 패키지에 위치(`MemberAddressService`), 컨트롤러는 `mypage` 패키지로 분리.

- `MemberAddressRepository`: `findByMember_SeqOrderByDefaultYnDesc()` — 기본 배송지(`default_yn='Y'`)가 항상 목록 최상단
- `DeliveryAddress` 엔티티에 setter 대신 의도가 명확한 메서드 정의:
  - `clearDefault()` → `defaultYn = "N"`
  - `setDefault()` → `defaultYn = "Y"`
  - `update(dto)` → 수정 필드 일괄 반영

**기본 배송지 단일 유지 로직**

DB에 UNIQUE 제약 없음 → 서비스 레벨에서 처리:
1. 새로 추가/수정 시 `defaultYn = "Y"`이면 기존 목록에서 Y인 것을 전부 N으로 변경 (`clearCurrentDefault`)
2. 이후 신규 항목 저장 또는 dirty checking으로 자동 UPDATE
3. `@Transactional` 안에서 엔티티 필드 변경 → JPA dirty checking으로 별도 save 불필요

**API 엔드포인트** (`/api/mypage/address`)

| 메서드 | 경로 | 역할 |
|--------|------|------|
| `GET` | `/{seq}` | 수정 모달 단건 조회 |
| `POST` | `/` | 배송지 추가 |
| `PUT` | `/{seq}` | 배송지 수정 |
| `PATCH` | `/{seq}/default` | 기본 배송지 변경 |
| `DELETE` | `/{seq}` | 배송지 삭제 |

모든 엔드포인트에서 `@AuthenticationPrincipal`로 받은 memberSeq와 경로의 seq를 대조해 본인 소유 검증.

**프론트 (`mypage/address.html`)**

- Thymeleaf `#strings.toUpperCase(addr.defaultYn)` 사용 — DB에 소문자로 저장된 값도 안전하게 비교
- 기본 배송지 뱃지는 `Y`일 때만, "기본 배송지로" 버튼은 `Y`가 아닐 때만 렌더링
- 카카오 우편번호 API 연동 모달 (추가/수정 공용)
- `id="address"` → `id="baseAddress"`로 변경 — 브라우저 내장 `window.address` 객체와 충돌 방지, JS body의 key는 `address:`로 유지

## 트러블슈팅 기록

| 문제 | 원인 | 해결 |
|------|------|------|
| `save()` 후 DB에 반영 안 됨 | 서비스 메서드에 `@Transactional` 누락 | `addCart()`, `addMyAddress()`에 추가 |
| `ChatRoomRepository` No property 'memberId' | 레포 메서드명이 엔티티 필드명(`memberSeq`)과 불일치 | `findByMemberId` → `findByMemberSeq` 수정 |
| `default_yn` 비교 오작동 | DB에 소문자 `'y'`로 저장된 데이터 존재 | Thymeleaf `#strings.toUpperCase()` 적용 + `UPDATE SET default_yn = UPPER(default_yn)` |
| JS `null` 오류 (`addressAlias`) | HTML `id="alias"` vs JS `getElementById('addressAlias')` 불일치 | `id="addressAlias"`로 통일 |
| Querydsl "Attempt to recreate Q class" | `src/main/generated/` 가 git 추적됨 | `.gitignore`에 추가 + `git rm -r --cached` |
